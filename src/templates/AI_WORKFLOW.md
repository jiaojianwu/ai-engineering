# Vibe Coding 工作流

## 场景判断（开发前必读）

**根据任务类型选择对应流程，禁止小题大做：**

| 场景                 | 判断标准                                     | 使用流程       |
| -------------------- | -------------------------------------------- | -------------- |
| 🔥 **小修复/小改动** | 修改文案、修复单个 bug、调整样式、改一个字段 | **快速流程**   |
| 🛠️ **中等任务**      | 新增/修改一个组件、一个 API 对接、一个小功能 | **标准流程**   |
| 🏗️ **大功能开发**    | 新增完整页面、复杂交互、多模块联动、架构调整 | **完整流程**   |
| 📝 **问答咨询**      | 仅解释代码、技术方案讨论、不涉及代码修改     | **无需工作流** |

---

## 流程一：快速流程（小修复/小改动）

**适用场景**：改文案、修 bug、调样式、改类型、删废弃代码等，预计耗时 < 30 分钟。

```
1. 读取 checklist.md
2. 读取 tech-stack.md（如需要）
3. 定位代码 → 修改 → 自检
4. 运行 tsc / lint 验证
5. 告知用户修改内容
```

**自检清单**：

- [ ] 修改范围最小化，不引入无关变更
- [ ] 符合 code-standards 规范
- [ ] tsc / lint 通过

---

## 流程二：标准流程（中等任务）

**适用场景**：新增/修改组件、对接 API、实现独立小功能，预计耗时 30 分钟 - 2 小时。

```
阶段1：设计（AI执行，1轮）
├── 功能分析 → 简要技术方案（口播或简短 markdown）
└── 人工确认（快速 yes/no）

阶段2：开发（AI执行）
├── 读取 checklist.md + code-standards.md
├── 按模块开发 + 实时自检
└── 更新相关 docs.md（如需要）

阶段3：交付（AI自检 + 人工抽查）
├── pnpm run tsc
├── pnpm run lint
└── 告知用户变更摘要
```

---

## 流程三：完整流程（大功能开发）

**适用场景**：新增页面、复杂功能、跨模块改动，预计耗时 > 2 小时。

```
阶段1：设计（AI并发）      阶段2：开发（AI执行）     阶段3：交付
┌─────────────────────┐  ┌─────────────────────┐  ┌────────────┐
│ 需求分析 + 技术设计   │→ │ 任务开发 + 自检      │→ │ 审核 + 推送 │
│      ↓              │  │      ↓              │  │            │
│   人工校正(1次)      │  │   人工Review(1次)   │  │            │
└─────────────────────┘  └─────────────────────┘  └────────────┘
```

### 阶段 1：设计（AI 并发执行）

**AI 并发执行**：

```
并行任务：
├── Task A: 功能分析 → 输出「功能拆解.md」
├── Task B: 技术设计 → 输出「技术设计方案.md」
└── Task C: 任务规划 → 输出「开发任务清单.md」
```

**AI 指令模板**：

```
基于需求文档，请完成以下设计：

1. 功能分析：
   - 识别核心/次要功能
   - 标注优先级（P0/P1/P2）
   - 列出验收标准

2. 技术设计：
   - 目录结构
   - 组件设计
   - 状态管理方案

3. 任务规划：
   - 按优先级分组（每任务 0.5-2h）
   - 标注依赖关系
   - 影响面评估

输出 3 个独立的 md 文件。
```

**人工校正（1次）**：

- 审核 3 个设计文档
- 统一反馈，一次性校正
- 确认后进入开发阶段

### 阶段 2：开发（AI 执行）

**开发前必须**：

- 遵循 `AGENTS.md` 中定义的强制步骤（读取 checklist.md、file-priority.md 等）
- 按需加载规则文件（tech-stack.md、code-standards.md 等）

**开发流程**：

```
for 每个任务 in 任务清单:
    1. 实现代码
    2. 自检（规范、边界、性能）
    3. 更新相关 docs.md（如需要）
    → 所有任务完成后统一提交
```

**自检清单**：

- [ ] 代码符合规范
- [ ] 命名清晰
- [ ] 边界情况处理

### 阶段 3：交付（人工审核）

**验证**：

```bash
pnpm run tsc
pnpm run lint
pnpm run build:web
```

**审核重点**：

- 业务逻辑正确性
- 整体代码质量
- 文档完整性

---

**修改说明文档**：当修改完成代码后，检查是否有说明文档，如果有更新文档，如果没有询问是否需要创建。创建说明文档参考 `.agents/skills/code-documentation-expert/SKILL.md`。
